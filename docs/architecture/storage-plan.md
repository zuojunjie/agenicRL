# AutoDL 存储规划

**目的**：在租用 AutoDL 训练 Search-R1 全流程（P0–P5）中**零数据丢失**，且避免每次开机重传/重装造成的钱和时间浪费。

> 此文档由 `autodl-scheduler` sub-agent 引用，是后续所有开机/关机/释放决策的依据。

---

## 0. 不可妥协的硬约束

| 约束 | 含义 | 后果 |
|---|---|---|
| **连续关机 ≥ 15 天 → 自动释放** | AutoDL 强制规则，无法关闭 | 系统盘 + 数据盘**所有数据**被清空，**不可恢复** |
| 镜像保存只覆盖**系统盘** | `/root/autodl-tmp` 数据盘**不被打包** | 仅靠镜像无法保住 50GB+ 的检索索引 |
| 镜像无法从外部导入 | 只能在 AutoDL 内创建 | 不要指望本地打包上传镜像 |
| 数据盘按容量计费（即使关机） | 容量定了就是定了 | 50GB 索引常驻 = 持续少量费用 |

---

## 1. 三层存储模型

把所有数据**强制分到这三层**，每层有不同的"保存"机制：

```
┌─────────────────────────────────────────────────────────────┐
│ L1 环境层（系统盘 / ~30 GB 默认）                            │
│   conda env / verl / vllm / cuda toolkit / apt 包 / 代码    │
│   → 保存机制：保存镜像                                       │
│   → 重建成本：30 分钟到 1 小时（装包慢）                     │
├─────────────────────────────────────────────────────────────┤
│ L2 大数据层（数据盘 /root/autodl-tmp / 50–200 GB）          │
│   Qwen2.5-3B 权重 / Wikipedia 索引 / NQ HotpotQA 数据集 /   │
│   训练 checkpoint                                            │
│   → 保存机制：关机保留 + 百度网盘冷备                        │
│   → 重建成本：几 GB ~几十 GB 的传输（贼慢）                  │
├─────────────────────────────────────────────────────────────┤
│ L3 代码 + 产物层（任意位置，但必须是 git 跟踪 / 网盘备份的）  │
│   训练脚本 / 实验配置 / 关键 ckpt / 训练日志 / wandb run     │
│   → 保存机制：git push（代码） + 百度网盘（ckpt/log）        │
│   → 重建成本：低（代码） / 中（ckpt 重传）                   │
└─────────────────────────────────────────────────────────────┘
```

**核心原则**：**任何一份重要数据至少存在两层里**。镜像挂了能从数据盘恢复；数据盘被释放能从百度网盘恢复；百度网盘被误删能从镜像 + git 重建。

## 2. 每种数据的处置表

| 数据 | 体积 | 主存位置 | 备份位置 | 备份频率 |
|---|---|---|---|---|
| conda env (verl, vllm, transformers...) | ~10 GB | 系统盘 | **镜像**（环境定型后立刻存） | 每次环境大改 |
| Qwen2.5-3B 权重 | ~6 GB | 数据盘 `/root/autodl-tmp/models/` | 百度网盘 | 一次（不变化） |
| Wikipedia 检索索引 | ~50 GB | 数据盘 `/root/autodl-tmp/wiki_index/` | 百度网盘 `tar.gz` | 一次（不变化） |
| NQ / HotpotQA / 2WikiMQA 数据集 | ~3 GB | 数据盘 `/root/autodl-tmp/data/` | 百度网盘 | 一次 |
| 训练脚本 / 配置 | KB | 系统盘（在 git repo 内） | **GitHub** + 镜像 | 每次改完 git push |
| 训练 checkpoint | 6–30 GB / 个 | 数据盘 `/root/autodl-tmp/ckpts/<run_id>/` | 百度网盘（仅关键 ckpt） | 训练结束立即上传 best/final |
| 训练日志 / wandb / tensorboard | MB–GB | 数据盘 `/root/autodl-tmp/logs/` | 百度网盘（每次训练后） | 训练结束 |
| 实验报告 / 笔记 | KB | git repo + **飞书**（已接入 lark-cli） | 飞书自带版本 | 写完即推 |

## 3. 生命周期 Playbook（按时间顺序）

### Phase A — 首次配环境（一次性）

1. 开个**最便宜的卡**（甚至无卡模式 ¥0/h）配环境，不要拿 4×A100 配环境
2. 装好 verl / vllm / transformers / 依赖
3. 跑通 1 步 smoke test 确认环境 OK
4. **立刻关机** → 控制台 → 更多操作 → **保存镜像** → 命名 `search-r1-env-v0`
5. 镜像保存完成后再决定是否释放实例（如果数据盘上还要存大数据，**不要释放**）

### Phase B — 大数据初始化（一次性）

1. 开机（无卡模式即可）
2. 把 **base model + Wikipedia 索引 + 数据集**下到数据盘 `/root/autodl-tmp/`
3. 立刻在 AutoDL 文件管理里 **`tar` 打包 → 上传百度网盘**作为冷备
4. 关机

### Phase C — 每次训练前（高频）

```
开机前检查清单 ☐
☐ 选用之前保存的镜像（不要从基础镜像重装）
☐ 数据盘已挂载且数据完整（ls /root/autodl-tmp 验证）
☐ git pull 拿到最新训练脚本
☐ 本次实验要存的 ckpt 路径已规划：/root/autodl-tmp/ckpts/<run_id>/
☐ 训练脚本末尾加自动关机：训练完跑 `/usr/bin/shutdown`
```

### Phase D — 训练中

- ckpt 直接写到数据盘（**绝不写系统盘**，否则 30 GB 系统盘秒满）
- 日志实时同步到 wandb（云端备份）
- 关键里程碑 ckpt（best / final）在训练脚本里 inline 上传到百度网盘

### Phase E — 训练后立即（在关机前！）

```
关机前检查清单 ☐
☐ 关键 ckpt（best / final）已传百度网盘？
☐ 训练日志 + wandb run id 已记录到 git？
☐ 实验结论 / 关键指标已写到飞书笔记？
☐ 系统盘有没有意外产物？（如果有，决定要不要刷新镜像）
☐ 距离上次开机超过 X 天？是否需要刷镜像保存新装的包？
```

### Phase F — 长期不用（>1 周不训）

- 数据盘 50 GB 持续收费（每天几毛），可以接受
- 但若**预计 ≥ 2 周不用**：
  1. 确认百度网盘备份**完整可恢复**（关键 ckpt 下载下来 md5 校验一次）
  2. 确认镜像存在
  3. 释放实例（省下硬盘费）
  4. 下次用：从镜像 + 百度网盘恢复（成本约 1–2 小时，可接受）

### Phase G — 每周硬性巡检（不管在不在训练）

```
每周一次（推荐周日）☐
☐ 控制台看实例状态：连续关机几天？> 12 天必须警觉
☐ 镜像还在不在？（防误删）
☐ 百度网盘容量还够不够？
☐ 上周新增数据是否进了备份？
```

## 4. 数据丢失场景与恢复路径

| 场景 | 触发条件 | 恢复路径 | 恢复成本 |
|---|---|---|---|
| 实例被 15 天自动释放 | 长时间没开机 | 新建实例 → 选保存的镜像 → 从百度网盘下载数据集 / ckpt | 1–2 小时 + 流量费 |
| 系统盘异常 | 罕见，AutoDL 故障 | 同上 | 1–2 小时 |
| 忘了保存镜像就释放 | 人为失误 | 新建实例 → 从基础镜像 + git 重装环境 → 从百度网盘下数据 | 半天 |
| 百度网盘文件误删 | 人为失误 | 镜像 + git + 数据盘三者其一恢复 | 看丢了什么 |
| 同时挂掉镜像 + 数据盘 + 百度网盘 | 极小概率 | git 里的代码 + HuggingFace 重新下基础模型 + 重建索引 | 数天 |

**第 5 行的场景**才是真正的灾难。三层备份就是为了避免这一行发生。

## 5. 容量与成本估算

| 项目 | 容量 | 月费（估） | 备注 |
|---|---|---|---|
| 系统盘（30 GB 默认，免费） | 30 GB | ¥0 | 装环境用 |
| 数据盘（建议扩到 100 GB） | 100 GB | ¥10–15 | 实例不在也收费 |
| 镜像（≤30 GB / 个） | 30 GB × 2 个版本 | 包含在数据盘费里或单独少量 | 只留 1–2 个版本 |
| 百度网盘 | 200 GB+ 推荐超级会员 | 用户已自备 | 冷备核心 |
| **存储总成本** | — | **≈ ¥15–20 / 月** | 训练 GPU 费用之外的 fixed cost |

## 6. 关键禁止项

- ❌ **不要**在 4×A100 高价机上做配环境、下数据这种 CPU 活
- ❌ **不要**把数据写到系统盘（会满 + 镜像变臃肿）
- ❌ **不要**指望"再过几天就用了不释放也行" — 一旦 14 天没碰，立刻刷开关机一次重置计时
- ❌ **不要**只存一份镜像 — 每次大改环境前先备份旧镜像（命名带版本号）
- ❌ **不要**忽略关机前检查清单 — 90% 的数据丢失发生在"我下次再说" 这句话之后

## 7. 工具引用

- 备份打包：AutoDL 控制台内置文件管理 → 选目录 → 打包 → 上传百度网盘
- 镜像保存：控制台 → 容器实例 → 关机 → 更多操作 → 保存镜像
- 自动关机：训练脚本末尾追加 `/usr/bin/shutdown` 或 AutoDL 提供的关机脚本
- 容量监控：AutoDL 控制台 / `df -h` / `du -sh /root/autodl-tmp/*`
- 笔记记录：`lark-cli`（已配置，见 [feishu_setup](../../.claude/agents/autodl-scheduler.md)）

## 8. 一页速记（贴墙版）

```
三层存储：环境 → 镜像；数据 → 数据盘 + 百度网盘；代码 → git + 飞书
三个绝不：绝不 4×A100 配环境；绝不数据写系统盘；绝不连续关机 14 天
三个必做：训练前装镜像；训练后传 ckpt；每周巡检一次
一条铁律：任何重要数据，至少存在两个地方
```

## 9. 实测验证记录（2026-04-26）

在无卡模式实例上完整走了一遍"植入 marker → 关机 → 保存镜像 → 释放 → 用镜像建新实例 → 验证"流程。结论：

**已验证为真**：
- ✅ 关机不丢系统盘和数据盘任何文件（md5 验证一致）
- ✅ 保存镜像的 UI 路径 = `更多 → 保存镜像`（关机状态下才能点）
- ✅ 镜像独立可恢复 — 释放原实例后，从镜像建的新实例完整复刻系统盘内容（包括手装的 conda env）
- ✅ 镜像确实**只覆盖系统盘** — 新实例的数据盘是全新的，原 datadisk marker 消失
- ✅ 跨容器（新 hostname）恢复无副作用 — 同区域同镜像，conda env 直接可用

**实测数据点**：
| 操作 | 耗时 | 备注 |
|---|---|---|
| 保存镜像（系统盘 ~700MB） | ~3 min | 真训练后系统盘 5–10G 时预计 10–20 min |
| 关机 → 重新开机 | < 1 min | AutoDL 关机本质是停容器，不重启宿主机 |
| 释放 → 新建实例（用已存镜像）→ SSH 可用 | < 1 min | 比预期快得多，"释放重建"成本可接受 |

**意外发现（已纳入 §6 禁止项）**：
- 数据盘设备名跨实例会变（`/dev/nvme1n1` → `/dev/md0`），但挂载点 `/root/autodl-tmp` 稳定。**代码里只用挂载点，不要 hardcode 设备路径**。
- 数据盘新建后并非全空，AutoDL 会自动放 `.autodl/` 目录。判断"数据盘是否首次使用"不能用"目录是否为空"。

**未覆盖的验证（待真实训练时再测）**：
- 跨**区域**镜像可用性（本次同区域 cqa1）
- 真实大体积系统盘（含 verl + vllm + cuda toolkit ~10GB）的镜像保存耗时
- 百度网盘 `tar` 上传/下载实际带宽
- 抢占式实例的镜像/数据盘行为
